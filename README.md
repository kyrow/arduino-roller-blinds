# ESP32 Roller Blind Controller (HTTP)

Контроллер рулонных штор на базе ESP32 с управлением через HTTP API. Поддерживает шаговый двигатель, синхронизацию времени по NTP, расписание автоматического открытия/закрытия, а также сохранение позиции в энергонезависимой памяти.

## Возможности

- Управление рулонной шторой через HTTP-запросы (`/open`, `/close`, `/status`).
- Точное позиционирование в процентах (0–100%) с использованием шагового двигателя через драйвер (DIR/STEP/EN).
- Плавное ускорение и торможение благодаря библиотеке [AccelStepper](https://github.com/waspinator/AccelStepper).
- Автоматическое расписание открытия и закрытия по времени суток.
- Синхронизация времени по NTP с ежедневной повторной синхронизацией в полночь.
- Сохранение текущей позиции в `Preferences` (NVS) — после перезагрузки штора «помнит», где остановилась.
- Автопереподключение к Wi-Fi при потере соединения.
- Защита от заклинивания: таймаут движения и принудительная остановка.

## Аппаратная часть

| Сигнал | GPIO ESP32 | Назначение |
|--------|-----------|------------|
| DIR    | 16        | Направление вращения |
| STEP   | 17        | Импульсы шагов |
| EN     | 4         | Включение драйвера (LOW = enable) |

Параметры мотора задаются в конструкторе `RollerBlind` в `src/main.cpp`:

```cpp
RollerBlind(16, 17, 4, /*timeoutMs*/ 1000000, /*stepsToTop*/ 56000,
            /*maxSpeed*/ 2500, /*acceleration*/ 1200);
```

- `stepsToTop` — количество шагов от полного закрытия (0%) до полного открытия (100%). Подбирается под конкретный механизм.
- `timeoutMs` — таймаут одного перемещения, после которого мотор аварийно останавливается.

## Сборка и прошивка

Проект использует [PlatformIO](https://platformio.org/).

```bash
# Сборка
pio run

# Прошивка
pio run --target upload

# Монитор последовательного порта (9600 baud)
pio device monitor -b 9600
```

Зависимости устанавливаются автоматически из `platformio.ini`:

```ini
[env:esp-32-rollblind]
platform = espressif32
board = esp32dev
framework = arduino
lib_deps =
  waspinator/AccelStepper @ ^1.64
```

## HTTP API

Сервер слушает порт **80**. Все ответы возвращаются в формате JSON.

### `GET /open`

Полностью открывает штору (по умолчанию). Принимает необязательный параметр `position` (0–100).

```http
GET /open                 # 100%
GET /open?position=50     # 50%
```

Ответ:
```json
{ "status": "ok", "action": "open", "position": 50 }
```

### `GET /close`

Полностью закрывает штору (позиция 0%).

```json
{ "status": "ok", "action": "close", "position": 0 }
```

### `GET /status`

Возвращает текущее состояние и позицию.

```json
{ "state": "stopped", "position": 42 }
```

Возможные значения `state`: `moving`, `open`, `closed`, `stopped`.

### `GET /time`

Возвращает текущее время устройства и параметры расписания.

```json
{
  "time": "2026-04-20 10:32:11",
  "hour": 10,
  "minute": 32,
  "schedule_enabled": true,
  "open_at": "8:00",
  "close_at": "18:00",
  "opened_today": true,
  "closed_today": false
}
```

Если время ещё не синхронизировано — ответ `500` с `{"error":"time not synced"}`.

### `GET /schedule`

Включает или выключает расписание во время работы.

```http
GET /schedule?enabled=true
GET /schedule?enabled=false
GET /schedule              # просто узнать текущее состояние
```

Ответ:
```json
{ "schedule_enabled": true }
```

> Параметр сохраняется только в RAM — после перезагрузки берётся значение `SCHEDULE_ENABLED` из `config.h`.

## Готовые запросы

В корне репозитория лежит файл `.http`, совместимый с расширением [REST Client для VSCode](https://marketplace.visualstudio.com/items?itemName=humao.rest-client). Создайте `.env` в корне проекта:

```

```

И отправляйте запросы прямо из редактора.

## Логика расписания

Каждую секунду в `loop()` вызывается `checkSchedule()`:

1. При смене дня (по `tm_yday`) сбрасываются флаги `openedToday` / `closedToday`.
2. Если текущее время совпадает с `SCHEDULE_OPEN_*` — штора открывается, флаг `openedToday` ставится в `true`.
3. Аналогично для закрытия.
4. Действия выполняются ровно один раз в сутки, чтобы повторное попадание в ту же минуту не запускало мотор снова.

Параллельно `checkDailyTimeSync()` ежедневно в 00:00 запрашивает NTP-сервер для компенсации дрейфа RTC.

## Сохранение состояния

Текущий процент открытия сохраняется в namespace `blinds-shared` (`Preferences`/NVS) при каждой остановке мотора. При следующем запуске позиция восстанавливается, чтобы система не считала, что штора находится в нуле.

## Защита и отказоустойчивость

- **Таймаут перемещения** — если за `timeoutMs` (по умолчанию 1 000 000 мс) мотор не достиг цели, движение прерывается и в Serial выводится `CRITICAL: ROLLER BLIND TIMEOUT!`.
- **Reconnect Wi-Fi** — при потере соединения каждые 5 секунд предпринимается попытка переподключения (до 20 попыток за раз).
- **Driver enable/disable** — `EN` пин активируется только во время движения, что снижает нагрев драйвера и потребление.

## Разработка

Логи устройства выводятся в Serial на скорости **9600**. Включают:
- состояние подключения к Wi-Fi (IP, MAC),
- результат NTP-синхронизации,
- каждый HTTP-запрос (`/open`, `/close`),
- запуск/остановку расписания,
- сохранение позиции в NVS,
- таймауты и ошибки.
